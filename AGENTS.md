# AGENTS.md

This is the canonical AI contributor guide for Sixxer. Read this first in every
Codex, Cursor, Claude Code, Gemini, or Copilot session, then load only the
specific files needed for the task.

## First-read map

Use this order instead of scanning the whole repo:

1. `AGENTS.md` - repository rules, architecture map, validation policy.
2. `docs/ai/context.md` - product/design/Linear workflow briefing for agents.
3. `docs/product/vision.md` - product direction and personas when feature scope
   or copy is involved.
4. `docs/product/backlog.md` - Linear roadmap index and original task seed; use
   it for original acceptance criteria, not live issue status.
5. `Design.md` - design system tokens and UX principles for UI changes.
6. Relevant implementation files from the maps below.

## Project summary

- Product: Sixxer.
- Domain: cricket club and team management.
- Audience: amateur cricket admins, captains, treasurers, players, and future
  parent/guardian users.
- Stack: TanStack Start, React 19, TypeScript, Tailwind CSS v4, shadcn/ui,
  Supabase, Playwright.
- Deployment target: Cloudflare-compatible runtime.
- Product posture: mobile-first operational app, not a marketing site.

## Non-negotiable rules

- Keep changes small and task-focused.
- Do not hand-edit `src/routeTree.gen.ts`; it is generated by TanStack Router.
- Do not commit `.env`, `.env.*`, `.dev.vars`, Supabase service role keys, or
  other secrets.
- Prefer existing patterns over new abstractions.
- Avoid broad formatting-only churn unless the task is explicitly formatting
  cleanup.
- Preserve role boundaries between admins/captains/treasurers/players.
- Treat Supabase RLS and server/client boundaries as part of the feature, not
  incidental plumbing.
- For UI work, follow `Design.md`: dark athletic minimalist, mobile-first,
  thumb-friendly, high-contrast, and operationally dense.

## Linear workflow

- The owner tracks implementation work in Linear.
- Linear is the live source of truth for issue status, prioritization, and
  current task scope.
- If a Linear issue is provided, treat its title, description, and acceptance
  criteria as the active scope.
- If no Linear issue is provided, use `docs/product/backlog.md` and
  `docs/product/vision.md` for product context, then keep the change narrowly
  scoped to the user request.
- When creating or refining tasks, use the backlog template:
  `Problem`, `User Story`, `Scope`, `Out Of Scope`, `Acceptance Criteria`,
  `Validation`.
- Mention any meaningful scope split back to the user instead of silently
  expanding one task into a broad feature.

## Architecture map

### Routing and app shell

- `src/router.tsx` - router creation.
- `src/routes/__root.tsx` - root route, providers, global shell wiring.
- `src/routes/_authenticated.tsx` - authenticated layout boundary.
- `src/components/AppShell.tsx` - main app shell.
- `src/components/BottomTabBar.tsx` - mobile bottom navigation.
- `src/routeTree.gen.ts` - generated; do not edit manually.

### Product routes

- `src/routes/_authenticated/home.tsx` - home dashboard.
- `src/routes/_authenticated/calendar.tsx` - cross-team calendar.
- `src/routes/_authenticated/groups.tsx` - groups list.
- `src/routes/_authenticated/groups.$teamId*.tsx` - team pages, tabs, settings,
  members, events, posts, polls, payments.
- `src/routes/_authenticated/events.*.tsx` - event creation and detail.
- `src/routes/_authenticated/payments.*.tsx` - payment creation and detail.
- `src/routes/_authenticated/profile.tsx` - profile.
- `src/routes/_authenticated/notifications.tsx` - notification center.
- `src/routes/login.tsx`, `src/routes/signup.tsx` - auth entry points.

### Feature hooks and domain logic

- `src/features/home/use-home-summary.ts` - home data.
- `src/features/events/*` - event detail/admin/series/weather logic.
- `src/features/calendar/*` - calendar feed and upcoming events.
- `src/features/teams/*` - team context, layout, members, user groups.
- `src/features/payments/use-payment-detail.ts` - payment detail data.
- `src/features/polls/use-team-polls.ts` - polls.
- `src/features/profile/use-profile.ts` - profile data.
- `src/features/notifications/use-notification-center.ts` - notifications.

### Platform and integrations

- `src/platform/*` - platform service abstractions.
- `src/integrations/supabase/client.ts` - browser Supabase client.
- `src/integrations/supabase/client.server.ts` - server Supabase client.
- `src/integrations/supabase/auth-middleware.ts` - auth middleware.
- `src/integrations/supabase/types.ts` - generated database types.

### Shared UI and utilities

- `src/components/ui/*` - shadcn/ui primitives.
- `src/components/EmptyState.tsx`, `Avatar.tsx`, `PageHeader.tsx` - shared

- `src/lib/auth.tsx`, `theme.tsx`, `format.ts`, `maps.ts`, `share.ts`,

```bash
npm run seed:dev

```

## Validation policy

- Prefer `npm run build` after meaningful code changes.
- Use focused Playwright specs for changed critical workflows when practical.
- Treat `npm run lint` as noisy for now because the repo has pre-existing
  formatting debt.
- For documentation/rules-only changes, validate by reading the changed files
  and checking git diff; a full build is not required.

## Test and seed notes

- E2E tests rely on a non-production Supabase project and the seed script.
- `scripts/seed-dev.mjs` provisions disposable test data including
  `admin@sixxer.test` and player accounts.
- Never point seed or E2E flows at production.

## AI handoff expectations

- Start by stating which context files you loaded.
- Before editing, identify the likely files to touch.
- Keep a short running note of assumptions when Linear context is missing.
- In final responses, include changed files, validation performed, and any
  remaining risks or follow-ups.
